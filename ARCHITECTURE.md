## Видение проекта

Браузерная тактическая RPG, вдохновлённая **Final Fantasy Tactics**, построенная на **Phaser 4 + TypeScript + Vite**.  
Внешних арт-ассетов нет — вся графика рисуется процедурно через Phaser Graphics API.

---

## Текущий статус: **Фаза 1 — Боевой цикл + Режим кампании**

### ✅ Готово
- Репозиторий: `kuirsan26-lab/fft-phaser`
- Scaffold: Phaser 4 + Vite + TypeScript
- Изометрический рендерер сетки (карта 10×8, 4 типа тайлов, высота 0–3)
- Менеджер ходов на основе CT (система Charge Time, как в FFT)
- BFS-поиск пути с учётом высоты и прыжка
- Боевая система: ближний бой, дальний, магия (Огонь / Лёд / Гром), лечение, AoE, стан
- ИИ: движется к врагам, выбирает лучшую способность
- UI: меню действий, панель порядка ходов, тултип юнита, лог боя, всплывающие числа урона
- 4 класса: Воин, Рыцарь, Маг, Лучник
- Переключатель автобоя: кнопка (верхний правый угол) + клавиша A
- **Режим кампании:** создание отряда → 10 битв → победа
  - `SquadCreationScene`: имя отряда (ввод с клавиатуры), выбор 4 классов кликом
  - `CampScene`: состав отряда, HP/MP переносятся между битвами, кнопка лечения, прогресс-бар
  - Перманентная смерть: погибшие юниты освобождают слот навсегда
  - Масштабирование врагов: множитель статов растёт с номером битвы (×0.7 → ×1.7)
  - Автонавигация после боя: победа → лагерь, поражение → создание отряда
- **Зона опасности врага:** при наведении на врага показывается оранжевая зона его максимальной дальности атаки
- **Фиксы рендеринга (2026-04-24):**
  - Юниты центрируются по визуальному центру тайла (`getUnitPosition`: `+TILE_H/2`)
  - `screenToGrid` использует корректные центры тайлов → клики работают точно
  - Оверлеи вставляются через `container.addAt` после базового тайла — возвышенный рельеф корректно их перекрывает
  - Оверлеи сортируются по `col + row` (алгоритм художника) перед отрисовкой

## Архитектура

```
src/
├── main.ts               — конфиг Phaser, регистрация сцен
├── scenes/
│   ├── BootScene.ts           — заставка → SquadCreationScene
│   ├── SquadCreationScene.ts  — имя отряда + выбор 4 классов, запуск кампании
│   ├── BattleScene.ts         — основной игровой цикл; поддерживает кампанию (принимает CampaignState)
│   ├── CampScene.ts           — лагерь между битвами: ростер, лечение, продолжение
│   └── UIScene.ts             — параллельная сцена: заголовок, ход, оверлей конца боя
├── systems/
│   ├── GridSystem.ts     — изометрический рендеринг, подсветка тайлов, математика координат
│   ├── TurnManager.ts    — очередь ходов на CT, управление юнитами
│   ├── Pathfinder.ts     — BFS-движение, расчёт дальности, восстановление пути
│   └── CombatSystem.ts   — формулы урона, выполнение способностей, ИИ
├── entities/
│   └── Unit.ts           — класс юнита: статы, HP/MP, эффекты статусов, ссылки на спрайты
├── data/
│   ├── UnitData.ts       — шаблоны классов, определения способностей
│   ├── MapData.ts        — карта высот тайлов, позиции спавна
│   └── Campaign.ts       — типы CampaignState, состав врагов по битве, масштабирование статов
└── ui/
    ├── ActionMenu.ts     — меню действий (Движение / Способности / Ожидание)
    ├── TurnOrderPanel.ts — полоса предстоящих ходов с CT-барами
    └── BattleLog.ts      — лог боя + всплывающие числа урона
```

## Ключевые архитектурные решения

### CT-система (Charge Time)
У каждого юнита `ct` растёт на `unit.spd` за тик. При `ct >= 100` юнит действует, затем `ct -= 100`. Более быстрые юниты ходят чаще — ощущение как в оригинальной FFT.

### Изометрическая проекция
```
screen_x = (col - row) * TILE_W/2   + offsetX
screen_y = (col + row) * TILE_H/2   + offsetY  - height * TILE_DEPTH
```
Тайлы рисуются сзади вперёд, отсортированные по `col + row` (алгоритм художника).  
Оверлеи (подсветка движения/атаки/опасности) вставляются в контейнер через `addAt` сразу после своего базового тайла — это гарантирует корректное перекрытие рельефом.

### Коммуникация между сценами
`BattleScene` испускает Phaser-события (`turnStart`, `battleOver`, `campaignInfo`). `UIScene` слушает их через `scene.get('BattleScene').events.on(...)`. Данные кампании передаются напрямую через `scene.start('SceneName', data)`.

### Архитектура ИИ
Приоритетный ИИ в `CombatSystem.aiChooseAction`:
1. Использовать не-атакующую способность на врага в зоне досягаемости
2. Иначе — ближний бой по самому слабому врагу в радиусе
3. Если никого нет в радиусе — двигаться к ближайшему врагу

Известное ограничение: все юниты ведут себя одинаково независимо от класса (маг бежит в ближний бой, рыцарь не прикрывает союзников).

### Формулы боя
- Физический урон: `(atk × 1.8 × множитель) - (def × 0.5) ± 0–5 случайно`
- Магический урон: `(mag × 2.5 × множитель) - (mag_цели × 0.3) ± 0–5 случайно`
- Лечение: `mag × 3 × множитель`

Высота пока не влияет на урон — запланировано добавить +15% при атаке сверху вниз.

## Принципы разработки (уроки индустрии)

Этот раздел — ориентир для будущих реализаций. Основан на анализе FFT, Fire Emblem, Into the Breach, Darkest Dungeon и паттернах Phaser.

### Прогрессия кампании

**Два слоя прогрессии** (лучшие roguelite-игры: Hades, Slay the Spire):
- **Внутри забега**: временные улучшения, сбрасываются при проигрыше — лечение, временные баффы
- **Мета-прогрессия**: постоянные разблокировки между кампаниями — новые классы, способности, реликвии

Текущая реализация имеет только первый слой. Мета-прогрессия — следующий ключевой шаг.

**Экономика лагеря** (по модели Darkest Dungeon):
- Ресурсы должны создавать значимые решения: лечить одного или копить на апгрейд?
- Бесконечный рекрутинг + дорогое снаряжение = потеря юнита ощутима, но не фатальна для кампании
- Скудость ресурсов создаёт напряжение; избыток убивает интерес

**Привязанность к юнитам** (Fire Emblem):
- Даже 1–2 строки уникального диалога/предыстории на юнита делают перманентную смерть эмоциональной
- Без эмоциональной привязки перманентная смерть — просто неудобство

### Дизайн карт и энкаунтеров

**Каждая битва — тактическая задача** (Into the Breach):
- Состав врагов: смешивать ближний бой + дальний + поддержку (интереснее, чем просто скейлить статы)
- Соотношение врагов: 1.5–2× от числа игровых юнитов для средней сложности
- Высота тайлов должна использоваться тактически: лучники на возвышении, маги в тылу
- Каждая из 10 битв должна иметь замысел — не просто "сильнее предыдущей"

**Масштабирование сложности**:
- Предпочтительнее: разнообразие состава врагов → сложность через роли, а не через голые статы
- Текущий подход (множитель ×0.7–1.7) — временное решение, заменить на ручной дизайн энкаунтеров

### AI

**Приоритеты улучшения AI** (сейчас слишком прямолинейный):
- Роль-ориентированное поведение: маги держат дистанцию (≥3 тайлов от врага), рыцари прикрывают магов
- Отступление: юнит с HP < 30% отходит вместо самоубийственной атаки
- Концентрация огня: добивать ослабленных вместо размазывания урона
- Фланкирование: атаковать с разных сторон (долгосрочно)

### Технические паттерны Phaser

**Управление состоянием**:
- `game.registry` — для глобальных данных (CampaignState), доступен из любой сцены без передачи через `scene.start(data)`
- `scene.data` — для данных конкретной сцены
- Переход на registry снизит связанность между сценами

**`sleep()` вместо `stop()`**:
- `scene.sleep()` замораживает сцену, сохраняя все объекты — дешевле stop+relaunch
- Использовать для UIScene: `sleep` при переходе в лагерь, `wake` при возврате в бой

**Жизненный цикл слушателей**:
- Всегда снимать `battleScene.events.off(...)` в `'shutdown'` — уже реализовано в UIScene, применять везде

**Пул объектов**:
- Числа урона, частицы, подсветки тайлов — создавать пул при старте сцены, переиспользовать
- Текущий подход (создать/уничтожить каждый раз) приведёт к просадкам при большом числе событий

### Фидбек и читаемость

**Каждое действие = цепочка отклика** (правило <100мс):
- Атака: анимация замаха → числа урона → тряска спрайта (есть)
- Смерть: должна быть выразительнее (уникальная анимация, реплика)
- Ход ИИ: паузы между решениями уже есть, увеличить при сложном AI

**Цветовое кодирование**:
- Сейчас: синий = движение, красный = атака, оранжевый = зона опасности врага
- Добавить: жёлтый = AOE-зона, зелёный = лечение, серый = недосягаемо
- HP-бары: зелёный >50%, жёлтый 25–50%, красный <25% (не реализовано)

### Доступность (минимум)
- Не полагаться только на цвет — добавить форму/иконку как второй канал
- Кнопки ≥44px по касаемой области (уже выдержано в ActionMenu)
- Возможность ремаппинга клавиш — актуально после добавления большего числа хоткеев
